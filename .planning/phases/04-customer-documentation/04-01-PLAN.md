```python
wire_machine_macros(machine)  # No profile, no overrides
```

```python
wire_machine_macros(machine, macro_overrides={
```

```python
wire_machine_macros(machine, macro_overrides={

}, strict=True)
```

  <action>
Create `tutorials/macro_customization.ipynb` with the following structure. All code must run without `qm.open()`, `qm.run()`, or `machine.connect()`.

1. **Title + intro (markdown):** Explain the four workflows and that the notebook runs without hardware.

2. **Cell: Workflow 1 — Use defaults**
   - Import `build_tutorial_machine`, `wire_machine_macros`
   - Build machine, add voltage points (or call a helper that does), `wire_machine_macros(machine)` with no overrides
   - Assert or print that `quantum_dots`, `quantum_dot_pairs`, `sensor_dots` have expected macros (initialize/measure/empty for dots and pairs; measure only for sensor)
   - Build a small `qua.program()` that calls `component.initialize()`, `component.measure()` etc. and assign to a variable (no run)

3. **Cell: Workflow 2 — Type-level override**
   - Define a minimal custom macro class (e.g. `CustomInitMacro`) with `@quam_dataclass`, inheriting from `InitializeStateMacro` or similar
   - `wire_machine_macros(machine, macro_overrides={"component_types": {"QuantumDot": {"macros": {"initialize": {"factory": CustomInitMacro, "params": {...}}}}}}, strict=True)`
   - Print/assert the override is applied

4. **Cell: Workflow 3 — Instance-level override**
   - Use `instances` key with path like `quantum_dots.<id>` or `qubits.<id>`
   - Override one macro for one instance
   - Print/assert the override is applied

5. **Cell: Workflow 4 — External macro package**
   - Create an inline "fake" external package: a `build_macro_overrides()` function in the notebook that returns a dict like workflow 2/3
   - Call `wire_machine_macros(machine, macro_overrides=build_macro_overrides(), strict=True)`
   - Explain that in practice this lives in a separate Python package

6. **Cell: @quam_dataclass anti-pattern**
   - Define `BadMacro` (no decorator) and `GoodMacro` (@quam_dataclass) — both minimal QuamMacro subclasses
   - Assign BadMacro to a component, call `machine.save(tmp_path)` — expect failure or demonstrate state loss (e.g. reload and show macro missing/broken)
   - Assign GoodMacro, save, load, assert macro survives
   - Add markdown explaining: "Custom macro classes must use @quam_dataclass for save/load round-trips."

Order of cells: intro → workflow 1 → 2 → 3 → 4 → @quam_dataclass. Use correct component types (QuantumDot, QuantumDotPair, SensorDot) throughout.
